### B1 — Tab 横向滑动与 WebView 竖向滚动手势冲突

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已修复 |
| **Commit** | `c2aefce` |
| **现象** | 浏览 Wiki 页面时竖向滑动必须非常垂直才能触发滚动，稍有横向偏移就会切换 Tab。而横向切换 Tab 很轻松 |
| **根因** | `TabBarView` 内部使用 `PageView` 风格的横向滑动手势 `GestureRecognizer`，与 WebView 的竖向滚动手势在 Flutter gesture arena 中形成竞态 |
| **修复** | 用 `IndexedStack` 替换 `TabBarView`，TabBarView 不再注册任何滑动手势。Tab 切换仅靠点击 `TabBar` 完成。移除 `AutomaticKeepAliveClientMixin`（IndexedStack 天然保持 child 存活） |
| **涉及文件** | `wiki_browser_page.dart` |

### B3 — 深色模式自动恢复、默认强制深色

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已修复 |
| **Commit** | `b09e470` |
| **现象** | 进入 App 后页面自动变深色；用户手动切回日间后，Tab 切换或站内链接刷新后又变回深色 |
| **根因** | 两重问题叠加：① `_isDarkMode = true` 默认开启深色；② `build()` 中的同步块（`if (_isDarkMode != theme.isDark)`）在每次重建时强行将 `_isDarkMode` 对齐到 `theme.isDark`（两套主题均返回 `true`）。用户手动切到日间后，下一次 `setState` 触发 rebuild → 同步块检测到 `false != true` → 立即强制重置为 `true` → 深色 CSS 重新注入 |
| **修复** | ① `_isDarkMode = false`（默认日间）；② 删除 build 中的同步块，让用户的手动切换成为唯一权威来源；③ 删除随即变为死代码的 `_toggleDarkModeFromTheme()` |
| **涉及文件** | `wiki_browser_page.dart` |

### B4 — 深色模式下图片失真（`invert(0.88)` 双重复合滤镜不互消）

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已修复 |
| **Commit** | `8cdd098` |
| **现象** | PRTS Wiki 和 Warfarin Wiki 的所有图片在深色/日间模式切换后颜色失真、对比度异常 |
| **根因** | CSS `filter` 是顺序叠加的渲染管线。`html { filter: invert(0.88) }` 反色整页后，`img { filter: invert(1) }` 在已反色的像素上再做一次完全反色，两次不能完美抵消。白色 (255)→224→31，黑色 (0)→31→224，对比度被严重压缩 |
| **修复** | `html { filter: invert(1) }`——100% 反色 + 100% 反向反色 = `invert(1) × invert(1)` 顺序叠加 = 恒等变换。`hue-rotate(180deg)` 在两层各出现一次 → 累计 360° = 色调无变化 |
| **涉及文件** | `wiki_dark_mode.dart` |
| **副作用** | 100% invert 比 88% 的黑白对比更锐利。如觉刺眼可后续加 `brightness(0.9)` 柔化 |

### B5 — PRTS 轮播海报图片仍有失真（`background:` 简写选择器缺失）

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已修复 |
| **Commit** | `080e589` |
| **现象** | 大部分图片修复后，PRTS 首页顶部的可翻页轮播海报图片仍然失真 |

| **修复** | CSS 新增 `[style*="background:url"]` 和 `[style*="background: url"]` 选择器 |
| **涉及文件** | `wiki_dark_mode.dart` |

### B6 — Warfarin Wiki 透明底白色图标在日间模式下不可见（class 背景图）

| 字段 | 内容 |
|------|------|
| **状态** | ❌ 未完全解决（原因见 B8） |
| **Commit** | `080e589`（部分修复）→ `191155d`（策略变更） |
| **现象** | Warfarin Wiki 中职业图标、元素图标等白色透明背景的图标在日间模式下难以看清 |
| **根因** | 图标使用 CSS class 设置背景图（如 `.skill-icon { background: url('sprite.png') -20px 0 }`），元素无 `style` attribute，CSS attribute 选择器无法命中。`html { filter: invert(1) }` 将白色图标反色为黑色，而 class 背景图标未被 re-invert |
| **修复（方案一）** | 增加 JS DOM 遍历 + `window.getComputedStyle().backgroundImage` 检测 + `data-arklores-inverted` 标记 + `remove()` 清理。但未解决 `bg-black/70` 容器反色后白图标在白底上的融合问题 |
| **修复（方案二）** | 见 B8 —— 改为利用 Warfarin Wiki 的原生日间模式 |

### B7 — Warfarin Wiki 文字出现黑边/变粗（`_reInvertBackgroundsJS` 对日间路径误执行）

| 字段 | 内容 |
|------|------|
| **状态** | ✅ 已修复 |
| **Commit** | `d26143e` → `efa92d3`（amend） |
| **现象** | Warfarin Wiki 日间模式下武器名等文字变得特别粗，出现黑色描边效果 |
| **根因** | `_reInvertBackgroundsJS` 在 `inject()` 中位于两条路径之外，无论是否注入 CSS inversion 都会执行。Warfarin Wiki 走 "移除 `class="dark"`" 路径后不需要任何 CSS 滤镜，但后续的 JS 仍对带计算背景图的元素施加 `filter: invert(1)`，导致这些容器内的文字也被反色/滤镜化 |
| **修复** | 将 `_reInvertBackgroundsJS` 移入 `else` 分支（仅对 CSS inversion 路径执行）。同时将 `_cleanupJS` 也分出两条独立路径 |
| **涉及文件** | `wiki_dark_mode.dart` |

### B8 — Warfarin Wiki 日间模式整体方案缺陷（`filter: invert` 无法处理半透明背景+前景图标的视觉混合）

| 字段 | 内容 |
|------|------|
| **状态** | ❌ **未完全解决**（用户将自行研究） |
| **Commit** | `191155d` |
| **现象** | Warfarin Wiki 切换日间模式后，职业图标（`<div class="bg-black/70"><img profession_icon></div>`）在白色半透明底上消失 |
| **根因** | CSS `filter: invert(1)` 无差别作用页面上所有像素。当一个容器有 `bg-black/70` 且内含白色图标时：`bg-black/70` → `rgba(255,255,255,0.7)`（白半透明），图标 img 被 re-invert 回白色 → 白图标在白底上看不见。CSS filter 无法区分"这个白色是背景经反色来的"还是"这个白色是图标原本的颜色"——滤镜作用于像素层，不是语义层 |
| **修复尝试** | 检测 `<html class="dark">`，走"移除 `class="dark"`、利用站点的 Tailwind baseline 日间模式渲染"路径。理论上应是更干净的解法，但实际 Warfarin Wiki 的 Tailwind 配置可能未对所有 `dark:` 变体提供对等的 baseline 样式，导致部分渲染不完整。用户判断仍有问题 |
| **解决状态** | 用户将自行研究 Warfarin Wiki 的 Tailwind 配置以彻底解决 |
